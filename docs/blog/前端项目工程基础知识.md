# 前端项目工程基础知识：包管理器、lock 文件与环境变量

> 本文整理了加入一个大型企业级前端项目时，围绕包管理器、lock 文件、环境变量配置等工程基础知识的系统梳理，适合有前端基础、正在向全栈或工程化方向拓展的开发者。

---

## 一、包管理器：npm、yarn、pnpm 的区别

### 它们是什么

npm、yarn、pnpm 都是**包管理器**，作用只有一个：帮你下载和管理项目依赖的第三方代码。

可以类比成外卖平台：你要用某个第三方库（要吃饭），不用自己去找源码（不用去餐厅），直接运行安装命令（叫外卖）就行。三者送的东西一样，但配送方式不同。

| | npm | yarn | pnpm |
|---|---|---|---|
| 出现时间 | 最早，Node.js 自带 | Facebook 造，解决 npm 早期慢的问题 | 最新，解决磁盘占用问题 |
| 速度 | 慢 | 快 | 最快 |
| 磁盘占用 | 每个项目单独存一份 | 每个项目单独存一份 | 所有项目共用一份，硬链接 |
| 现在推荐 | 不推荐新项目用 | 还行 | 推荐 |

### 命令对比

```bash
npm install       # 安装依赖
yarn              # 安装依赖
pnpm install      # 安装依赖

npm run dev       # 跑项目（必须加 run）
yarn dev          # 跑项目（可省略 run）
pnpm dev          # 跑项目（可省略 run）
```

### 一个项目只能用一种包管理器

混用会产生多余的锁文件，导致团队成员装出来的依赖版本不一致。加入项目时，先看 README 或者问老成员用哪个，然后统一用那一个。

## 二、lock 文件是干什么的

### 问题根源

`package.json` 里的依赖版本是**模糊的**：

```json
{
  "dependencies": {
    "element-plus": "^2.3.0"
  }
}
```

`^2.3.0` 的意思是"2.3.0 或更新的版本都行"。

这带来一个问题：

```
你今天 install → 装了 element-plus 2.3.0
同事下周 install → 装了 element-plus 2.4.1（出了新版）

同一个 package.json，装出来的版本不一样
→ 你这里没问题，同事那里样式乱了
```

### lock 文件的作用

lock 文件（`package-lock.json` / `yarn.lock` / `pnpm-lock.yaml`）记录的是**精确版本**：

```
element-plus: 2.3.0（就是这个，不是别的）
以及它所有子依赖的精确版本
```

所有人 install 时读 lock 文件，装出来的依赖 100% 一致。

### 三种 lock 文件对应关系

| 包管理器 | lock 文件 |
|---|---|
| npm | package-lock.json |
| yarn | yarn.lock |
| pnpm | pnpm-lock.yaml |

**lock 文件是自动生成的**，不需要手动维护，运行 install 命令时自动产生或更新。一个项目里只应该有一种 lock 文件，多了说明团队混用了包管理器。

### lock 文件什么时候起作用

```
pnpm install  → 读 lock 文件，决定装哪个精确版本  ✅
pnpm dev      → 和 lock 文件无关，只是跑已经装好的东西
```

## 三、`npm run dev` 的完整执行流程

### 标准的 scripts 字段

`package.json` 里的 scripts 是命令的快捷方式，标准写法如下：

```json
{
  "scripts": {
    "dev":   "vite",
    "build": "vite build",
    "preview": "vite preview"
  }
}
```

执行 `pnpm dev` 时，实际是找到 scripts 里的 `dev` 字段，然后执行对应的命令。

### `node_modules/.bin/` 是什么

安装 vite 时，包管理器不只是把源码放进 `node_modules/vite/`，还会在 `node_modules/.bin/` 里放一个快捷方式：

```
node_modules/
├── .bin/
│   ├── vite          ← 快捷方式，指向 vite 的可执行文件
│   ├── vue-tsc
│   └── eslint
├── vite/
│   └── ...           ← vite 真正的代码
```

完整执行流程：

```
pnpm dev
↓
scripts: { "dev": "vite" }
↓
去 node_modules/.bin/ 找 vite
↓
.bin/vite 指向 node_modules/vite/bin/vite.js
↓
Node.js 执行这个文件
↓
开发服务器启动
```

为什么要这么绕？因为 `vite` 不是全局安装的，只存在于这个项目的 `node_modules` 里。直接在终端输 `vite` 会报 `command not found`，但通过 `pnpm dev` 执行时，pnpm 会自动把 `node_modules/.bin/` 加进查找路径。

### 特例：scripts 里还能自定义更多东西

scripts 里不只能写工具命令，也可以自定义任意 shell 命令，甚至调用另一个包管理器。

以某个实际项目为例：

```json
{
  "scripts": {
    "i":   "pnpm install",
    "dev": "vite --mode env.local"
  }
}
```

这里出现了一个容易混淆的情况：

| 命令 | 实际执行 | 用的包管理器 |
|---|---|---|
| `npm i` | npm install（npm 内置缩写） | npm |
| `npm run i` | pnpm install（读 scripts 里的 "i"） | pnpm |
| `pnpm i` | pnpm install | pnpm |

**`npm i` 和 `npm run i` 是完全不同的东西。** `npm i` 是 npm 内置的 install 缩写，和 scripts 无关；`npm run i` 才是去 scripts 里找 `"i"` 这个命令。

如果项目用的是 pnpm，应该用 `pnpm i` 而不是 `npm i`，否则会生成多余的 `package-lock.json`。

## 四、Vite 环境变量的完整机制

### 标准的多环境设计

一个标准的 Vite 项目，scripts 通常长这样：

```json
{
  "scripts": {
    "dev":   "vite --mode dev",
    "test":  "vite build --mode test",
    "stage": "vite build --mode stage",
    "prod":  "vite build --mode prod"
  }
}
```

对应的环境变量文件：

```
.env           → 公共配置，所有环境共享，提交 git
.env.local     → 本地私有覆盖，不提交 git
.env.dev       → 开发环境专属配置，提交 git
.env.test      → 测试环境专属配置，提交 git
.env.stage     → 预发布环境专属配置，提交 git
.env.prod      → 生产环境专属配置，提交 git
```

### 文件命名规则

```
.env                  # 所有 mode 都读，放公共配置
.env.local            # 所有 mode 都读，本地私有，不提交 git
.env.[mode]           # 只有指定 mode 时才读，团队共享
.env.[mode].local     # 只有指定 mode 时才读，本地私有
```

### `--mode` 参数的作用

`--mode` 是 Vite 内置参数，后面跟什么字符串，就去读对应的 `.env.[mode]` 文件：

```bash
vite --mode dev   → 读 .env + .env.dev + .env.local
vite --mode test  → 读 .env + .env.test + .env.local
vite --mode prod  → 读 .env + .env.prod + .env.local
```

注意 `.env.local` 比较特殊，**任何 mode 下都会被读取**。

### 多文件覆盖逻辑

优先级从低到高，后读的覆盖先读的同名变量：

```
.env  →  .env.[mode]  →  .env.local  →  .env.[mode].local
低                                                        高
```

以 `pnpm dev`（即 `--mode dev`）为例：

```bash
# .env（所有环境共享）
VITE_APP_TITLE=细胞生产系统
VITE_TIMEOUT=10000

# .env.dev（开发环境，git 里有，团队共享）
VITE_API_URL=http://dev-server:8080
VITE_ENV=dev

# .env.local（你个人本地，git 里没有）
VITE_API_URL=http://localhost:3000   ← 覆盖 .env.dev 里的同名变量
```

最终生效：

```
VITE_APP_TITLE = 细胞生产系统           ← 来自 .env
VITE_TIMEOUT   = 10000                  ← 来自 .env
VITE_ENV       = dev                    ← 来自 .env.dev
VITE_API_URL   = http://localhost:3000  ← .env.local 覆盖了 .env.dev
```

### `.env.local` 的核心价值

它是**你的私人草稿纸**，解决了团队协作中改配置互相影响的问题：

```
没有 .env.local 时：
你本地调试需要改 .env.dev 里的 API 地址
→ 不小心提交了 git
→ 同事拉代码，API 地址变成你的 localhost:3000
→ 同事接口全部报错

有了 .env.local：
你改自己的 .env.local，不进 git
→ 只影响你自己，同事完全感知不到
```

### 特例：不按标准来的项目

理解了标准之后，再看一些项目的特殊设计就容易理解了。

比如某个项目的 scripts 是这样写的：

```json
{
  "scripts": {
    "dev": "vite --mode env.local"
  }
}
```

`--mode env.local` 是一个自定义的 mode 名，Vite 会去找 `.env.env.local` 这个文件。如果这个文件不存在（大多数情况下确实不存在），就只读 `.env` 和 `.env.local`。

这其实是把标准做法里 `.env.dev` 和 `.env.local` 两层**合并成了一层**——本地开发的所有配置直接写进 `.env.local`，更简单，不需要两个文件叠加。

这个项目同时还有 `.env.dev`、`.env.test`、`.env.prod` 等文件，但它们不是给本地 `pnpm dev` 用的，而是用于打包部署到各个服务器时读取的。

这说明 `--mode` 后面跟什么完全由项目自己决定，没有强制规范，理解了标准机制，遇到任何变体都能看懂。

## 总结

| 概念 | 一句话 |
|---|---|
| 包管理器 | 帮你下载依赖，一个项目只用一种 |
| lock 文件 | 记录精确版本，保证团队依赖一致，自动生成不用手动维护 |
| `npm run dev` | 找 scripts 里的命令执行，从 node_modules/.bin/ 找工具 |
| `--mode` | 告诉 Vite 读哪个 `.env.[mode]` 文件，后面跟什么都行 |
| `.env.local` | 你的私人草稿纸，任何 mode 下都生效，不进 git |
